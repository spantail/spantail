---
title: 移行
description: 既存システムの作業記録とワークフローを Spantail へ移行する。
---

インスタンスが動き出したら、これまでの履歴も持ち込めます。既存の**作業記録**は
JSONL ファイルから [CLI](/ja/guides/tools/cli/) で一括移行できます。**エージェントの活動**は
インポート対象ではありません — エージェントを接続した時点から記録が流れ始めます。

## 事前準備

1. 移行先の[ワークスペースとプロジェクト](/ja/admin/projects/)を作成します。インポートは
   プロジェクトを**スラッグ**で解決するだけで、作成はしません。未知のスラッグがあると実行は
   失敗し、存在するスラッグの一覧が表示されます。
2. 作業記録の所有者となるユーザーで [CLI をインストールしてサインイン](/ja/guides/tools/cli/#インストール)
   します。インポートされた作業記録の作成者は、その認証情報のアカウントになります。人ごとの
   履歴は、その人ごとに 1 回ずつインポートしてください。

## JSONL へエクスポートする

移行元システムのエクスポートを、**1 行 1 つの JSON オブジェクト**からなるファイルに変換します。

```json
{"project":"website","entryDate":"2024-07-15","durationMinutes":90,"description":"オンボーディングフローをレビュー","tags":["review"],"externalId":"legacy-4711"}
```

`entryDate` は必須で、値のまま使われます — 作成者のタイムゾーンにおけるローカル日付であり、
変換は行われません。`project` はスラッグで、`--project` フラグでまかなえる行では省略できます。
全フィールドは [CLI の一括インポート（JSONL）](/ja/guides/tools/cli/) を参照してください。

### 移行元の ID を保持する

`externalId` に移行元システムでの ID を指定すると、それが Spantail 側のエントリ ID になり、
インポートが**冪等**になります。同じファイルを再実行しても、重複ではなく更新になります。
移行を予行演習し、エクスポートを直して再実行できるのはこのためです。`externalId` のない行は
実行するたびに重複します。

ID はインスタンス全体でユニークで、使える文字は `A-Za-z0-9._:-` です。移行元の裸の数値は
そのまま使わず、プレフィックスを付けてください（`4711` → `legacy-4711`）。

## 予行演習してからインポートする

まずは何も書き込まずに、ファイルの検証とプロジェクトスラッグの解決だけを行います。

```bash
spantail entries import work-entries.jsonl --workspace acme --dry-run
```

最初のリクエストの前にファイル全体が検証されるため、不正な行があれば行番号付きで失敗し、
何もインポートされません。予行演習が通ったらフラグを外します。

```bash
spantail entries import work-entries.jsonl --workspace acme
```

エントリはアトミックなバッチで送信されるため、各バッチは全件成功か全件失敗のどちらかです。
途中のバッチが失敗した場合は、エクスポートを直して同じファイルを再実行してください。
`externalId` を指定していれば、すでに登録済みのエントリは重複せず更新されます。

## エージェントの活動

エージェントのセッションは発生した時点で捕捉されるため、一括インポートはありません。
[エージェント活動の捕捉](/ja/guides/capturing-agents/)に従ってエージェントをインスタンスへ
向けるか、独自のツールから[エージェント ingest API](/ja/api/agent-ingest/) へ直接送信してください。
